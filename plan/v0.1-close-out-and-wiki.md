# v0.1 close-out — remaining gaps plus the /wiki knowledge surface

The six foundation lanes and both functional gap fixes (G3, G4) are merged.
This plan sequences what stands between here and a shipped v0.1 — the three
remaining rehearsal gaps — and adds one new lane: **G7, the Wiki**, the public
reading surface for the docs/ knowledge base, served at `/wiki`.

Decisions already made (2026-07-18): `/wiki` path on the main site, not a
subdomain. Generator: docmd (MIT, audited, pinned version). The Toolkit
fixture is untouched — the Wiki serves docs/, which today has no public
surface at all.

## The queue

| # | Lane | What ships | Blocked by |
|---|------|-----------|------------|
| 1 | **G2 — Deploy** | Worker + site live; `PUBLIC_AGENT_API` set at build | Infra decision (Azure vs Cloudflare — both configs exist) |
| 2 | **G5 — External run** | The `curationsdev/community` runbook executed for real; first non-self-audit run-record | G2 (intake live) or manual path today |
| 3 | **G6 — Metric hook** | First-fix rate measurement per `docs/audits/measurement-spec.md` | G2 (needs live events) |
| 4 | **G7 — Wiki** | docs/ served at `/wiki` via docmd, rebuilt on every docs change | Nothing — can run in parallel |
| 5 | **Live rehearsal** | The stranger gate re-run by a real person, fresh account | G2 + G5 |

G7 has no dependency on the gate work and can land while G2 decisions are
pending. Nothing in this queue may slip the live rehearsal — if the Wiki
competes for attention with gate work, the gate wins.

## G2 — Deploy (the one human decision)

Everything else in this plan is agent-executable; this lane starts with a
choice only Wyatt makes:

- **Where the worker lives** — `agent-worker/` carries both `wrangler.toml`
  (Cloudflare) and `Dockerfile.azure` + `infra/` Bicep (Azure). One becomes
  production; the other stays dormant.
- **Where the site builds** — `catalog-site/` needs `PUBLIC_AGENT_API` set to
  the worker's public origin at build time.

Exit proof: `/submit/project/` completes a real intake round-trip against the
live worker; `/api/audit/records` serves over the public origin; no silent
fallback anywhere.

Execute the existing `docs/audits/external-byoc-test.md` runbook from the

## G6 — First-fix measurement hook

Implement the smallest honest version of `docs/audits/measurement-spec.md`:
count (a) first-time audits, (b) re-runs with lineage, (c) re-runs whose delta
fixed at least one check. First-fix rate = c / a, target ≥60%.

Exit proof: the three counters increment correctly in a staged walkthrough;
no personal data beyond what the receipt store already holds.

## G7 — Wiki at /wiki

**What it is.** The docs/ corpus — PRDs, glossary, roadmap, stranger guide,
audit records (13 files, ~2,500 lines, growing fast) — rendered as a
searchable, navigable reading surface at `curations.dev/wiki`. This is the
layman ↔ docs ↔ expert bridge: one canonical place to answer "where do I
turn for…" without opening the repository.

**How it ships (one PR).**

1. docmd pinned at an exact version (npm `@docmd/core`, no floating tags) —
   supply-chain condition from the 2026-07-18 capability audit.
2. A build step (`catalog-site` CI + local script) that renders docs/ into
   the site's `/wiki` path with `baseUrl: /wiki` config; output is committed
   to nothing — generated at build time, like the rest of the site.
3. Docs frontmatter pass only where docmd needs it (title/description);
   content text is not rewritten in this lane.
4. Navigation link from the Board header to `/wiki`, styled as an exit to a
   subordinate surface — the Wiki is not the Board and does not imitate it.
5. The generated `llms.txt` and OKF knowledge bundles ship with the build —
   they are the substrate for the AI layer, which is **not** in this lane.

**Falsifying proof.** Delete a docs/ file locally and rebuild: the page and
its nav entry must vanish — the Wiki renders the repository's truth, never a
stale copy. And: no page of the Wiki may claim Board identity (no Board nav,
no run-record rows inside `/wiki`).

**Boundaries (from the capability audit).**

- The Visual Oracle governs the Board; the Wiki is a distinct subordinate
  surface and may use docmd's own theme.
- The AI layer — PRD-change pings, trend digests, "where to turn for" Q&A on
  Azure OpenAI (Microsoft for Startups credits) — is **v0.2 scope**. It gets
  its own PRD note before any code. The Wiki lane only guarantees the
  machine-readable substrate exists.
- docmd stays quarantined to the `/wiki` build step: no plugins with network
  access, no telemetry, version bumps by reviewed PR only.
